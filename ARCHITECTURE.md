# mind-clone — Устав архитектуры и эволюции

Самодостаточный документ. Любой агент/чат продолжает разработку, опираясь только на него.
Это инварианты, законы и векторы — НЕ история обсуждения. Не релитигировать отвергнутое (раздел 6).

---

## 1. АКСИОМЫ (инварианты — всегда истинны)

A1. **Изоляция.** Каждая фаза/роль работает в собственном контексте; данные между ними
    ходят только через явный артефакт на диске, не через общую память.
A2. **Разделение ответственности.** Одна роль = один контракт. Ретривер только извлекает,
    аналитик только судит. Роль не делает чужую работу.
A3. **LLM никогда не читает сырое хранилище.** Ни таблицу, ни jsonl напрямую. Только через
    команды, описанные в манифесте.
A4. **Доказательство артефактом, не памятью.** Любой вывод обязан цитировать row id,
    физически присутствующие в доставленном артефакте. Нет id → нет утверждения.
    Enforced границей контекста (у аналитика нет доступа к таблице), а не правилом в промпте.
A5. **Манифест — производная схемы, не параллельный документ.** Генерируется из живой БД
    интроспекцией. Рассинхрон невозможен by construction.
A6. **Скармливается трансформированный срез под интент, не блоб.** Полная таблица в контекст
    LLM не загружается никогда.
A7. **Истина в таблицах.** Не обсуждается.
A8. **No fabrication / label inference / report gaps.** (Базовые гардрейлы скилла, поверх всего.)
A9. **Retrieval ≠ selection.** `regex`/`fts`/keyword — это **механизм извлечения** (сеть, recall),
    НЕ критерий выбора. Совпадение по слову не является основанием для решения — это не
    семантическая работа. Аналитик обязан **прочитать доставленный контент** (тела целиком или
    разбитыми на чанки) и только тогда выбрать. Запрещено обосновывать выбор фактом «совпал
    keyword/regex» или одними заголовками. Ключевой поиск лишь приносит кандидатов; решает чтение.

## 2. ЗАФИКСИРОВАННЫЙ СТЕК

S1. Движок таблиц: **SQLite** через `sqlite3` из Python stdlib. Ноль установки,
    `CHECK`-enum + FK + NOT NULL + транзакции + FTS5. Один `clone.db` на клон.
S2. Слой доступа = **контур `loop.py`** (stdlib, единственный интерфейс) — drill-down
    `intent`/`sources`/`map`/`select`/`compile`, плюс `fts`/`get`/`import`/`render`/`verify`.
    Каждая команда возвращает срез + подсказку следующей; неизвестная → громкая ошибка.
S3. Запись: **`import`** — валидация на `INSERT` в транзакции (CHECK kind / FK / NOT NULL).
    Битые строки отклоняются движком с причиной и логируются, не ловятся постфактум.
S4. Манифест: **`render manifest`** генерирует контракт из живой БД на лету (JIT) — не
    хранимый файл, дрейф невозможен by construction (A5).
S5. Git: `clone.db` — рабочий артефакт; коммитится детерминированный
    `sqlite3 .dump → clone.sql` как diffable-зеркало.
S6. Язык: все персистентные артефакты — на английском; диалог — на языке пользователя.

## 3. МОДЕЛЬ ДАННЫХ (факты схемы)

D1. `sources` — карточка источника (метаданные, без текста). PK `id` (s001…).
D2. `evidence` — атомарные единицы мышления. PK `id` (eN_NNN), FK `source → sources.id`.
    Поля: date, kind, topic, context, quote (дословно), note (перевод + аналитика «что раскрывает»).
D3. `kind` — enum[14]: belief, argument, prediction, heuristic, anecdote, reasoning,
    framework, antipattern, mental-model, stance, axiom, causal, analogy, observation.
D4. `evidence_fts` — FTS5 над quote+note.
D5. Полный сырой транскрипт — отдельно в `raw/`, по `video_id`; в таблицу не дублируется.
D6. Связь 1:N — один источник → много evidence. Текст не повторяется.
D7. Известный долг: `video_id`/`date` в evidence избыточны (выводимы через FK) — кандидаты
    на нормализацию. `summary` в sources обрывается на полуслове — баг харвестера.

## 4. ЗАКОН МАНИФЕСТА (контракт LLM ↔ таблица)

M1. Манифест содержит: schema_hash + user_version (привязка к версии), поля и законы,
    enum + живое распределение, покрытие (даты, FTS), каталог команд, рецепты intent→команда,
    правила поиска, гардрейлы.
M2. Любое изменение таблицы (DDL/enum/миграция) в той же операции регенерирует манифест.
M3. Каждая команда печатает текущий schema_version → LLM всегда видит актуальность.
M4. Перед доверием манифесту при позднем открытии прогона — `manifest --verify`.

---

## 5. ПРОТОКОЛ ДВУХ РОЛЕЙ (цикл интент → ответ)

Артефакт прогона: `clones/<slug>/runs/<run-id>/{intent.md, retrieval.md, choice.md}`.

P1. **Вызов A — РЕТРИВЕР.** Контракт: по семантике интента и манифеста выбрать/написать
    команды, доставить **тела** (не только заголовки) в `retrieval.md`. `regex`/`fts` — лишь сеть
    для кандидатов (recall), не выбор. Не анализирует. Не делает выводов.
P2. **Вызов B — АНАЛИТИК.** Контракт: видит ТОЛЬКО артефакт (не таблицу, не свою память).
    **Обязан прочитать весь доставленный контент** — тела целиком или разбитыми на чанки — и
    только тогда выбрать (см. A9). Запрещено решать по факту keyword/regex-совпадения или по
    заголовкам. Пишет `choice.md` с цитатами row id, confidence, что отвергнуто и почему.
P3. **Шов = артефакт.** Единственный интерфейс между ролями.
P4. **Петля INSUFFICIENT.** Если аналитику не хватает данных — он не лезет в таблицу, а
    возвращает сигнал `INSUFFICIENT: нужно X` → новый раунд ретрива. Изоляция не нарушается.
P5. **Развязка стоимости.** Подбор команд (дорого) персистится в `retrieval.md` и
    переиспользуется; пересмотр вывода переписывает только `choice.md`, не гоняя ретрив заново.
P6. **Привязка к версии.** `intent.md` хранит schema_hash таблицы, против которой планировали;
    при позднем возврате расхождение хэша → пометить артефакт «планировался против старой схемы».
P7. `intent.md` — сырой + нормализованный интент. `retrieval.md` — команды + rationale +
    доставленный срез. `choice.md` — вывод, переписываемый.

## 6. ОТВЕРГНУТЫЕ АЛЬТЕРНАТИВЫ (не релитигировать)

R1. **TSV/CSV как истина** — отвергнуто. Многострочные free-text цитаты рушат строки;
    экранирование = самопальная CSV-библиотека хуже стандартной. Строго хуже даже jsonl.
R2. **jsonl как канон** — отвергнуто в пользу таблиц (A7). jsonl остаётся лишь как
    промежуточный формат эмита/импорта и diff-зеркало при желании.
R3. **DuckDB** — отвергнуто. Наш паттерн OLTP (точечные срезы + целостность), не OLAP.
    На сотнях строк колоночная скорость = ноль выгоды; добавляет бинарь, слабее CHECK/FK.
R4. **Грузить всю таблицу в контекст** — отвергнуто. ~130–170К токенов на сессию,
    деградация внимания «иголка в стоге», цена и латентность. Противоречит A6.
R5. **Двухуровневое чтение (headers/bodies) как часть архитектуры** — отвергнуто. Это в
    лучшем случае необязательная токен-оптимизация ВНУТРИ ретривера, к разделению ролей
    отношения не имеет. Изоляция проходит между вызовами (P1/P2), не внутри одного.
R6. **Манифест, написанный руками** — отвергнуто. Дрейфует. Только генерация из БД (A5).
R7. **Сервер БД (Postgres и т.п.)** — отвергнуто. Скилл self-contained, клоны едут в проект.

## 8. ГЛОССАРИЙ АРТЕФАКТОВ

- `clone.db` — таблицы-истина (SQLite): `meta` + `ep` (+ `ep_fts`). Один центр.
- `loop.py` — **контур**, единственный интерфейс: read (drill-down) + write (`import`) + `render` + `verify`.
- `render manifest` — контракт LLM↔таблица, генерируется из БД на лету (не файл).
- `runs/<id>/intent.md · selection.tsv · delivery.md · answer.md · log.jsonl` — прогон цикла, артефакты на диске.
- `render brain` / `render playbook` / `render sources` — «мозг» / метод / реестр как JIT-вью (не хранимые файлы).
- `config.json` — состояние (chat_language, варианты имени, заметки о пробелах).
